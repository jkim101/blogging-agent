## 1. 시스템 개요

이 프로젝트는 **다중 에이전트 파이프라인**으로, 다양한 소스(URL, PDF, YouTube 영상)를 입력받아 **한국어 + 영어 이중 언어 블로그 포스트**를 자동으로 생성한다.

### 핵심 특징

| 특징 | 설명 |
|------|------|
| **한국어 우선 전략** | 한국어 초고 작성 → 사실 검증 + 비평 → 영어로 번역. LLM 비용 ~50% 절감 |
| **사람 개입 (HITL)** | 두 지점에서 사람이 검토: 아웃라인 승인 + 발행 승인 |
| **Critic 재작성 루프** | 최대 3회까지 자동 재작성. 3회차엔 관대한 기준 적용 |
| **상태 영속화** | SQLite 기반 LangGraph 체크포인트. 중단 후 재개 가능 |
| **웹 + CLI 이중 인터페이스** | FastAPI 웹 대시보드 또는 터미널에서 실행 |

### 사용 기술 스택

```
LLM:          Claude API (Anthropic SDK) — tool_use + Pydantic 파싱
오케스트레이션: LangGraph — 그래프 기반 상태 머신, HITL 인터럽트
상태 저장:     LangGraph SqliteSaver (thread_id 기반 체크포인트)
웹 UI:         FastAPI + Jinja2 + HTMX
파싱:          Trafilatura (URL), PyMuPDF (PDF), youtube-transcript-api
빌드:          Hatchling
```

## 3. 파이프라인 흐름

```
[입력: URL / PDF / YouTube]
        ↓
Research Planner (Opus 4.6)
  → 소스 분석, 인사이트 추출, 블로그 아웃라인 생성
        ↓
[🧑 HITL #1: 아웃라인 검토] ── REJECT → END
        ↓ APPROVE (또는 수정 후 승인)
Writer (Sonnet 4.5)
  → 한국어 초고 작성
        ↓
Fact Checker (Sonnet 4.5)
  → 주장 사실 검증, 심각도별 이슈 분류 (HIGH/MEDIUM/LOW)
        ↓
Critic (Opus 4.6)
  → 품질 평가 (점수 1-10), PASS/FAIL 판정
        ↓
    ┌── PASS (점수≥7 AND 고심각도 이슈 없음)
    │       ↓
    │   Translator (Sonnet 4.5)
    │     → 한국어 → 영어 번역 (언어=ko-only면 건너뜀)
    │       ↓
    │   Editor (Sonnet 4.5)
    │     → 한국어 + 영어 각각 스타일/톤 다듬기
    │       ↓
    │   SEO Optimizer (Sonnet 4.5)
    │     → 한국어 + 영어 각각 메타데이터 최적화
    │       ↓
    │   [🧑 HITL #2: 발행 검토] ── REJECT → END
    │       ↓ APPROVE
    │   Publish
    │     → GitHub Pages + output/ 로컬 저장
    │
    └── FAIL (점수<7 OR 고심각도 이슈 존재)
            ↓ (재작성 횟수 < 3회면)
        Writer (재작성) ← Critic 피드백 + Fact Check 이슈 반영
            ↓ (Fact Checker로 돌아감)

        [3회 모두 실패 → 강제 PASS → Translator로 진행]
```

### 노드 목록 (LangGraph)

| 순번 | 노드명 | 역할 |
|------|--------|------|
| 1 | `research_planner_node` | 소스 분석, 아웃라인 생성 |
| 2 | `outline_review_node` | HITL #1 인터럽트 체크포인트 |
| 3 | `writer_node` | 한국어 초고 작성/재작성 |
| 4 | `fact_checker_node` | 사실 검증 |
| 5 | `critic_node` | 품질 평가 |
| 6 | `translator_node` | 한→영 번역 |
| 7 | `editor_node` | 스타일 편집 |
| 8 | `seo_optimizer_node` | SEO 최적화 |
| 9 | `publish_review_node` | HITL #2 인터럽트 체크포인트 |
| 10 | `publish_node` | 발행 |

### 조건부 엣지 (라우팅 로직)

| 엣지 | 조건 | 목적지 |
|------|------|--------|
| `route_after_outline_review` | REJECT | END |
| | APPROVE / EDIT | `writer_node` |
| `route_after_critic` | PASS | `translator_node` (또는 ko-only면 `editor_node`) |
| | FAIL + 재작성<3회 | `writer_node` |
| | FAIL + 재작성=3회 | 강제 PASS → `translator_node` |
| `route_after_publish_review` | REJECT | END |
| | APPROVE | `publish_node` |

## 4. 핵심 설계 결정

### 4.1 한국어 우선 전략 (Korean-First)

**문제**: 한국어 + 영어를 각각 처음부터 작성하면 LLM 비용이 두 배.

**해결**: 한국어로만 초고 작성 → 사실 검증 + 비평 → 영어로 번역.
- 사실 검증은 한국어 초고에만 실행. 결과를 영어 버전에도 공유.
- 비용 절감: ~50%

### 4.2 HITL 두 지점

- **아웃라인 승인**: 글쓰기 전에 방향성 확인. 잘못된 방향으로 전체 파이프라인이 돌아가는 낭비 방지.
- **발행 승인**: 최종 품질 게이트. 사람이 완성된 포스트를 보고 발행 여부 결정.

### 4.3 Critic 재작성 루프 (최대 3회)

- **1~2회차**: 일반 기준 (점수 ≥ 7, 고심각도 팩트 이슈 없음)
- **3회차**: 관대한 기준 (점수 < 5 이거나 구조가 이해 불가능한 경우만 FAIL)
- 3회 후에는 강제 PASS → 무한 루프 방지

### 4.4 Tool-Use + Pydantic 파싱

- 모든 구조화 출력은 Claude의 `tool_use` 블록 사용
- Pydantic 모델로 응답 검증
- 검증 실패 시 LLM에게 수정 요청 메시지와 함께 재시도

### 4.5 프롬프트와 로직 분리

- 시스템 프롬프트: `prompts/` 디렉터리
- 에이전트 로직: `agents/` 디렉터리
- 프롬프트만 바꿔도 에이전트 동작 수정 가능

### 4.6 SEO 옵티마이저 제약

SEO Optimizer는 본문 텍스트를 절대 수정하지 않는다. 오직:
- 제목 (최대 60자)
- 메타 설명 (최대 160자)
- 키워드 목록
- URL 슬러그
- 헤딩 구조

만 최적화한다.

#### `core/graph.py` — LangGraph 파이프라인 그래프

LangGraph를 사용해 파이프라인을 방향성 그래프(DAG)로 구성한다.

**핵심 구성 요소**:

```python
# SQLite 체크포인터 초기화
checkpointer = SqliteSaver.from_conn_string(SQLITE_DB_PATH)

# StateGraph 정의
workflow = StateGraph(PipelineState)

# 노드 추가 (각 에이전트 함수를 노드로 등록)
workflow.add_node("research_planner", research_planner_node)
workflow.add_node("outline_review", outline_review_node)
# ... 10개 노드 추가

# 엣지 연결
workflow.set_entry_point("research_planner")
workflow.add_edge("research_planner", "outline_review")

# HITL 인터럽트 설정
workflow.add_node("outline_review", outline_review_node)
# interrupt_before=["outline_review"] 로 그래프 컴파일 시 설정

# 조건부 엣지
workflow.add_conditional_edges(
    "critic",
    route_after_critic,  # 라우팅 함수
    {
        "writer": "writer",
        "translator": "translator",
        "editor": "editor",
    }
)

# 그래프 컴파일 (HITL 인터럽트 포인트 지정)
graph = workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["outline_review", "publish_review"]
)
```

**노드 함수 구조** (각 노드는 동일한 패턴):

```python
def writer_node(state: PipelineState) -> dict:
    agent = WriterAgent()
    result = agent.run(state)
    return {
        "draft_ko": result.draft_ko,
        "rewrite_count": state.get("rewrite_count", 0) + 1
    }
    # 반환값은 PipelineState에 병합(merge)됨
```

**라우팅 함수**:

```python
def route_after_critic(state: PipelineState) -> str:
    feedback = state["critic_feedback"]
    rewrite_count = state.get("rewrite_count", 0)

    if feedback.verdict == Verdict.PASS:
        if state["blog_config"].language == "ko-only":
            return "editor"
        return "translator"

    # FAIL 처리
    if rewrite_count >= MAX_REWRITE_ATTEMPTS:
        # 3회 초과 → 강제 통과
        return "translator"

    return "writer"  # 재작성
```

```python
        - 전체 구조와 섹션 순서 유지

        입력: edited_draft_ko (편집 완료된 한국어 포스트)
        출력: draft_en (영어 초고)
        도구 사용 없음 — plain text 출력
        """
```

```python
    - 톤 선택
    - 언어 선택 (ko+en / ko-only)
    - 대상 독자
    - 주 키워드
    - TL;DR 포함 여부
    """

def run_pipeline(args) -> None:
    """
    파이프라인을 실행하고 HITL 루프를 처리한다.

    1. 소스 파싱 (URL/PDF/YouTube 자동 감지)
    2. 블로그 설정 입력 (--quick이면 기본값)
    3. PipelineRunner.start() 호출
    4. HITL 루프:
       while not complete:
           status = runner.get_status(thread_id)
           if is_interrupted:
               if interrupt_type == "outline_review":
                   _review_outline(state)
               else:
                   _review_publish(state)
               runner.resume(thread_id, human_input)
    5. _print_results(state) — 결과 출력 및 파일 저장
    """

def _review_outline(state: PipelineState) -> dict:
    """
    터미널에서 아웃라인을 보여주고 승인/거절을 받는다.
    출력: {"decision": "approve"/"reject", "notes": "..."}
    """

def _review_publish(state: PipelineState) -> dict:
    """
    터미널에서 최종 포스트를 보여주고 발행 결정을 받는다.
    출력: {"decision": "approve"/"reject", "targets": [...]}
    """

def _print_results(state: PipelineState) -> None:
    """
    파이프라인 완료 후 요약 정보를 출력한다:
    - Critic 점수
    - 팩트체크 정확도
    - 재작성 횟수
    - 저장된 파일 경로 목록
    save_posts()를 호출해 output/ 에 파일 저장
    """
```

| Translator | Sonnet 4.5 | 번역 품질은 Sonnet으로 충분. 비용 절감 |
| Editor | Sonnet 4.5 | 문장 수준 편집. 대규모 추론 불필요 |
| SEO Optimizer | Sonnet 4.5 | 메타데이터 생성은 단순 구조화 출력 |

## 9. 비용 구조 분석

**단일 파이프라인 실행 시 예상 토큰 사용량**:

| 단계 | 입력 토큰 | 출력 토큰 | 소계 |

| Research Planner | ~1,500 | ~500 | 2,000 |

| Fact Checker (최대 3회) | ~1,500 × 3 | ~800 × 3 | 6,900 |
| Critic (최대 3회) | ~1,000 × 3 | ~500 × 3 | 4,500 |
| Translator | ~2,000 | ~2,000 | 4,000 |
| Editor (한/영 각 1회) | ~2,000 × 2 | ~2,000 × 2 | 8,000 |
| SEO Optimizer (한/영 각 1회) | ~1,000 × 2 | ~300 × 2 | 2,600 |
| **합계** | | | **~40,000** |

**예상 비용** (재작성 없이 1회 성공 기준):
- Opus (Research Planner + Critic): ~$0.03

- **총합: ~$0.08 / 파이프라인**

**한국어 우선 전략 절감 효과**:
- 한/영 각각 작성 시: 약 두 배 비용
- 번역 전략 사용 시: Translator 추가 (~$0.01) 만 소요
- **절감액: ~50%**

## 10. 배포 및 운영

### 로컬 개발

```bash
# 1. 패키지 설치 (개발 의존성 포함)
pip install -e ".[dev]"

# 2. 환경 변수 설정
cp .env.example .env
# .env 파일에 ANTHROPIC_API_KEY 등 입력

# 3. 테스트 실행
pytest

# 4. 웹 대시보드 시작
python main.py serve
# → http://localhost:8000

# 5. CLI 실행
python main.py run https://example.com
```

### Docker / Railway 배포

```bash
# Docker 빌드
docker build -t blogging-agent .

# Docker 실행 (data/ 볼륨 마운트)
docker run -p 8000:8000 \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/output:/app/output \
  -e ANTHROPIC_API_KEY=sk-ant-... \
  -e DASHBOARD_PASSWORD=mypassword \
  blogging-agent
```

```
output/
├── ai-trend-2026_ko.md     # 한국어 포스트
```
